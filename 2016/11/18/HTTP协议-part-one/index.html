1.从背景开始——Web及网络基础

当我们在浏览器(Web browser)的地址栏中输入URL时，我们的Web界面显示出了从某处得到的界面，这是如何实现的？

Web界面当然不是凭空出现的，它是根据我们指定的URL，浏览器(即客户端(client))从Web服务器端(server)获得的文件资源等信息，再将他们显示出来的.

这里，Web也就使用了我们的HTTP(HyperText Transfer Protocol, 超文本传输(转移?)协议)来完成从客户端到服务器端等一系列运作流程。可以说，Web是建立在HTTP协议上通信的。

HTTP是TCP/IP协议族的一个子集，因此，我们先来看一下TCP/IP协议：

FTP         HTTP       IEEE 802.3
UDP      DNS        IP        FDDI
PPPoE          TCP       ICMP
....

TCP/IP是互联网相关的各类协议族的总称.

TCP/IP协议族中重要的一点就是分层 也就是以下四层：

使用TCP/IP协议族进行网络通信时，会通过分层顺序与对方通信，发送端从上到下，接收端反之。

接下来我们分别来看看IP，TCP和DNS这三个协议。
简单的说，IP协议负责传输，TCP协议确保可靠性，DNS协议负责域名解析
(域名也就是我们常输的网址啦! e.g: www.example.com)

**IP**

IP协议的作用是把数据包传送给对方，这其中有不可避免的涉及到了MAC地址和IP地址。

IP协议和IP地址不是一个东西!
MAC地址(全名: Media Access Control Address)是网卡所属的固定地址
IP地址指明了节点被分配到的地址
IP地址是可更改的，而MAC地址基本上不会更改。

IP间的通信依赖MAC地址，通信的双方经过多台计算机和中转设备才能连接到对方，为了能够定位到对方，就需要MAC地址和解析地址的ARP(Address Resolution Protocol)，根据IP地址来反查出对应的MAC地址。

就象是这样：

Sender--->(192.0.43.10)[ARP]----->>---Router1
|                           (MAC:00-XX-C6-6B-XX-XX)
|                                      |
|                                      |
(192.0.43.10)                          |
Receiver----------<<------------------Router2(MAC:00-XX-B5-A5-XX-XX)  
(MAC:00-XX-A6-6B-XX-XX)

**TCP**

TCP协议位于传输层，提供字节流服务并且确保数据能到达目标。

字节流服务是什么呢？

为了方便大段数据的传输，字节流服务(Byte Stream Service)将他们分割成了以报文段(segment)为单位的数据包来管理。

前面说过TCP协议负责确保可靠性，将大文件分割是其一，而更重要的是，TCP协议能够确认数据送达与否。

先看图示：

|       |------>>------|        |
|       |     SYN      |        |
|       |              |        |
| Sender|------<<------|Receiver|
|       |   SYN/ACK    |        |
|       |              |        |
|       |------>>------|        |     
|       |     ACK      |        |

这就是TCP的三次握手策略(three-way handshaking)，用TCP协议把数据包送出去后，它一定会确认数据送达与否，使用flag—-即SYN(synchronize),ACK(acknowlegement).若在握手的某个阶段莫名中断，TCP会再次以相同的顺序发送相同的数据包。

**DNS**

为了访问对方计算机，用户通常使用主机名或域名，而不是通过IP地址(一大串的数字也没人愿意去记)，对于计算机而言，它并不擅长理解名称，但却对数字敏感，因此，DNS服务，就为此而生，负责通过域名查找IP地址，或逆向反差。

TCP协议将其分割

并把每个报文段可靠的传给对方

***Step 3.***

IP协议通过路由器等设备一边搜索对方的地址 一边中转 一边发送

***Step 4.Server接收到数据后***

TCP协议接受每个接收到的报文段

最后我们再来看一下URI和URl.

相比URI，我们更熟悉URL(Uniform Resource Locator，统一资源定位符)，也就是像这样的网页地址(http://example.com)

而URI(Uniform Resource Identifier 统一资源标识符)，它用字符串来标识某一互联网资源，而URL标识资源的地点，所以可以说URL是URI的子集。

通过几个URL例子可以更好理解这个：

ftp://ftp.is.co.za/rfc/rfc1808.txt
http://www.iefg.com/rfc/rfc1002.txt
telnet://192.0.2.16:80/

在HTML文档中做引用的时候，我们会看到这样的URL字符串: “/image/1.jpg”. 我们将这样的URL叫做相对URL，它是从浏览器中基本URI处指定的URL。
一个绝对URL(URI)形如这样：

http://user:pass@www.example.jp:80/dir/index.htm?uid=1#ch1

我们来一个一个部分来分解它 【标记(^)的是可选项】

http:或https:等*****协议           protocol
user:pass*****登录信息(^)          userinfo
www.…….jp*****服务器地址           host
//也可以是192.168.1.1(IPv4) 
//或[0:0:0:0:0:0:0:1](IPv6)
80*****服务器端口号(^)              port
/dir/index.htm***目录/文件         path
?uid=1*****查询字符串(^)           query 
#ch1*****片段标识符(^)             fragment
